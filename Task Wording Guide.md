# Task Wording Guide

Good task wording makes the dashboard, cleanup review, and AI outputs all work better.

## Good Task Wording

Use tasks that are:

- concrete
- outcome-based
- bounded
- distinguishable from similar work

Good examples:

- `Rewrite the Sleep And Notes card description to match the current UI.`
- `Add artifact-type frontmatter to generated markdown files.`
- `Review stale projects and choose one to revive this week.`
- `Draft AI Guardrails note with tone, planning rules, and avoidance rules.`
- `Move Obsidian DASH project references out of the Master Task Hub.`

Why these work:

- they describe an observable outcome
- they are easier to estimate
- they are easier to mark complete
- they produce cleaner AI suggestions

## Weak Task Wording

Avoid tasks that are:

- vague
- too broad
- really just categories
- not actionable on their own

Weak examples:

- `Work on plugin`
- `Fix dashboard`
- `Documentation`
- `Health stuff`
- `Catch up`

Why these fail:

- they hide the next step
- they are hard to prioritize
- they create noisy cleanup suggestions
- they are hard for AI to reason about

## Improve A Weak Task

Instead of:

- `Documentation`

Use:

- `Write a What Goes Where guide for hub, project notes, and AI context notes.`

Instead of:

- `Fix dashboard`

Use:

- `Remove stale Top 3 language from reminders and README.`

Instead of:

- `Health stuff`

Use:

- `Review symptom entries from this week and note recurring triggers.`

## Good Optional Annotations

If a task still feels too loose, add one of these:

- due date
- blocker reason
- unblock date
- estimate
- related project
- minimum step in the task text or note

## Best Practice

If a task cannot be finished or meaningfully advanced in one sitting, the wording is usually still too broad.
